01 — Problem Statement

Traditional IoT falls short.

Every existing motion capture system ties intelligence to hardware. When the model changes, you reflash firmware. When hardware changes, you rewrite code. This is broken.

🔒
Firmware Locked
Traditional embedded systems require complete firmware reflashing for every model update — taking 10–30 minutes per device, making real-time adaptation impossible.
🧠
Limited On-Device Memory
Edge devices like ESP32-C3 have <400KB RAM — nowhere near enough to run BiLSTM, optical flow, or any meaningful ML model locally.
📡
No Multi-Modal Fusion
IMU-only systems are blind to spatial context. Camera-only systems lack kinematic signals. No standard exists to fuse heterogeneous streams without custom middleware.
🌐
Multi-Network Reality
Real-world deployments span private LANs (wearables) and public WiFi (surveillance cameras) simultaneously — no existing framework handles this natively.
👁️
Occlusion Problem
A single camera loses the subject when occluded. IMU continues tracking even when visual is lost — but no system currently exploits this complementarity.
⚡
Update Agility Gap
Server-side intelligence update: <60 seconds. Firmware OTA cycle: 10–30 minutes. This 10-30x agility gap is the core thesis argument for MCP.

05 — Real-World Scenario

One person. Five sensors. Total awareness.

A person is observed simultaneously from multiple angles through a network of heterogeneous cameras — wearable, fixed, depth-sensing, and surveillance — all feeding a single intelligent server.

01

Subject wears IMU wristband

ESP32-C3 + MPU6050 streams 50Hz accelerometer + gyroscope data over private WiFi hotspot (Sweet_Dumpling) to port 8765 as JSON.

02

Fixed cameras observe externally

Nicla Vision (tripod, port 8766) captures 5fps JPEG frames. Two LILIN cameras on poles provide zone-detection coverage across the performance space.

03

MCP server fuses all streams

4 listener threads handle simultaneous data. BiLSTM classifies: rotating / walking / stationary. Zone detection adds spatial context (which 3×3 grid cell).

04

Cross-modal validation

Gyroscope gz (IMU) is correlated with optical flow magnitude (camera) using Pearson r. Target r > 0.7 confirms both sensors agree on rotation.

09 — Development Journey

From zero to fully live.

PHASE 1 — Architecture Design
Defining the Heterogeneous MCP Architecture
Identified the core problem: firmware-locked edge AI. Designed dual-network topology separating private IMU from public camera networks. Chose FastMCP as server framework.
PHASE 2 — Hardware Setup
5 Devices Wired, Flashed, and Connected
ESP32-C3 firmware flashed. Nicla Vision MicroPython single-loop pattern deployed (no _thread). LILIN cameras configured. OAK-D Lite DepthAI pipeline validated.

PHASE 4 — MATLAB Analysis
BiLSTM Training + Optical Flow Complete
BiLSTM trained (60 epochs, Adam, bilstmLayer(100)). Optical flow computed. Gyroscope magnitude plots generated for all 3 classes. Confusion matrix produced.

Full system running. Pearson r cross-modal correlation being measured. Final evaluation metrics being captured for thesis submission.
